<Note>
The sessions dataset stores aggregated session data used for release health monitoring in Sentry. It tracks session counts, crash rates, and user adoption across releases.
</Note>

## Overview

The sessions dataset is a specialized dataset focused on release health metrics. Sessions represent user interactions with an application and are the foundation for calculating crash-free rates, adoption, and release quality metrics.

<Note>
**Important**: As of recent Snuba versions, session data is increasingly being migrated to the generic metrics dataset. Check your Snuba configuration to determine which dataset is active for session metrics in your deployment.
</Note>

### Key Characteristics

- **Purpose**: Release health monitoring and crash analytics
- **Data Model**: Pre-aggregated session counts by time bucket
- **Primary Use Cases**: Crash-free rate calculation, session counts, user adoption
- **Related Datasets**: Works alongside events dataset for detailed error analysis

## Session Lifecycle

A session represents a single user's interaction with your application:

1. **Session Start**: User opens the application or resumes from background
2. **Session Active**: User interacts with the application
3. **Session End**: Application closes, crashes, or enters background

### Session States

<CardGroup cols={2}>
  <Card title="Healthy" icon="circle-check">

  <Card title="Errored" icon="triangle-exclamation">

## Data Model

Session data is typically stored in aggregated form:

### Core Dimensions

```yaml
org_id: UInt64             # Organization identifier
project_id: UInt64         # Project identifier (required)
release: String            # Release version
environment: String        # Environment name
timestamp: DateTime        # Time bucket
```

### Session Metrics

```yaml
sessions: UInt64           # Total session count
sessions_crashed: UInt64   # Crashed session count
sessions_errored: UInt64   # Errored session count
sessions_abnormal: UInt64  # Abnormal session count
users: UInt64              # Unique user count (or HLL aggregate)
users_crashed: UInt64      # Users who experienced crashes
users_errored: UInt64      # Users who experienced errors
```

### Additional Attributes

```yaml
duration_quantiles: Array(Float64)  # Session duration percentiles
duration_avg: Float64               # Average session duration
started: DateTime                   # Session start timestamp
```

## Release Health Metrics

The primary use case for sessions data is calculating release health metrics:

### Crash-Free Session Rate

Percentage of sessions that did not crash:

```
crash_free_session_rate = (sessions - sessions_crashed) / sessions * 100
```

### Crash-Free User Rate

Percentage of users who did not experience a crash:

```
crash_free_user_rate = (users - users_crashed) / users * 100
```

### Adoption Rate

Percentage of users on a specific release:

```
adoption_rate = users_on_release / total_users * 100
```

## Example Queries

<Note>
The exact query syntax depends on whether sessions data is in a dedicated sessions dataset or in the generic metrics dataset. These examples show the conceptual approach.
</Note>

### Crash-free session rate by release

```sql
SELECT 
  release,
  (sum(sessions) - sum(sessions_crashed)) / sum(sessions) * 100 as crash_free_rate
WHERE project_id = 1
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
GROUP BY release
ORDER BY crash_free_rate DESC
```

### Session count over time

```sql
SELECT 
  toStartOfHour(timestamp) as hour,
  sum(sessions) as total_sessions,
  sum(sessions_crashed) as crashed_sessions
WHERE project_id = 1
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
GROUP BY hour
ORDER BY hour
```

### Release adoption

```sql
SELECT 
  release,
  uniq(users) as unique_users,
  uniq(users) / (SELECT uniq(users) FROM sessions WHERE project_id = 1) * 100 as adoption_pct
WHERE project_id = 1
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
GROUP BY release
ORDER BY unique_users DESC
```

### User impact by environment

```sql
SELECT 
  environment,
  sum(sessions) as total_sessions,
  sum(sessions_crashed) as crashed,
  sum(users_crashed) as affected_users
WHERE project_id = 1
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
GROUP BY environment
ORDER BY affected_users DESC
```

### Session duration analysis

```sql
SELECT 
  release,
  quantile(0.5)(duration_quantiles) as median_duration,
  quantile(0.95)(duration_quantiles) as p95_duration,
  avg(duration_avg) as avg_duration
WHERE project_id = 1
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
GROUP BY release
ORDER BY median_duration DESC
```

## Integration with Events

Session data complements the events dataset:

1. **Sessions** provide high-level health metrics (crash rates, adoption)
2. **Events** provide detailed crash information (stack traces, context)

### Correlation Example

```sql
-- Find releases with high crash rates
WITH crash_rates AS (
  SELECT 
    release,
    (1 - sum(sessions_crashed) / sum(sessions)) * 100 as crash_free_rate
  FROM sessions
  WHERE project_id = 1
    AND timestamp >= toDateTime('2024-01-01 00:00:00')
    AND timestamp < toDateTime('2024-01-02 00:00:00')
  GROUP BY release
  HAVING crash_free_rate < 99.0
)
-- Then query events dataset for crash details
SELECT 
  e.group_id,
  e.message,
  count() as event_count
FROM events e
WHERE e.project_id = 1
  AND e.release IN (SELECT release FROM crash_rates)
  AND e.level = 'fatal'
GROUP BY e.group_id, e.message
ORDER BY event_count DESC
```

## Migration to Generic Metrics

Many Snuba deployments are migrating session data to the generic metrics dataset:

### Benefits of Generic Metrics

- **Unified Storage**: All metrics in one place
- **Better Performance**: Optimized for time-series queries
- **Flexible Granularity**: Multiple time bucket sizes
- **Tag Support**: Rich filtering and grouping

### Querying Sessions in Generic Metrics

When sessions are stored as metrics:

```sql
MATCH (metrics_counters)
SELECT 
  sumMerge(value) as session_count
WHERE org_id = 1
  AND project_id = 2
  AND metric_id = hash('session.count')  -- Metric ID for sessions
  AND timestamp >= toDateTime('2024-01-01 00:00:00')
  AND timestamp < toDateTime('2024-01-02 00:00:00')
  AND tags[hash('release')] = hash('1.0.0')
```

## Data Ingestion

Sessions are typically ingested from SDK session tracking:

### SDK Session Envelope

SDKs send session envelopes with aggregated session data:

```json
{
  "init": true,
  "started": "2024-01-01T12:00:00Z",
  "duration": 300.5,
  "status": "exited",
  "errors": 2,
  "release": "1.0.0",
  "environment": "production",
  "distinct_id": "user-123"
}
```

### Aggregation Pipeline

1. **SDK**: Tracks session lifecycle
2. **Relay**: Forwards session envelopes
3. **Sentry**: Aggregates sessions by time bucket
4. **Kafka**: Publishes aggregated data
5. **Snuba**: Ingests and stores metrics

## Best Practices

<AccordionGroup>
  <Accordion title="Set session timeouts appropriately">
    Configure SDK session timeouts based on your application type:
    - Mobile apps: 30 seconds to 5 minutes
    - Web apps: 1-5 minutes
    - Background tasks: Custom based on expected duration
  </Accordion>
  
  <Accordion title="Track all session states">
    Don't just track crashes - monitor errored and abnormal sessions too for a complete picture.
  </Accordion>
  
  <Accordion title="Use release and environment consistently">
    Ensure consistent release and environment tagging across your organization for accurate metrics.
  </Accordion>
  
  <Accordion title="Combine with events for debugging">
    Use session metrics to identify problems, then drill into events dataset for details.
  </Accordion>
  
  <Accordion title="Set up alerts on crash-free rates">
    Alert when crash-free rates drop below acceptable thresholds (typically 99%+).
  </Accordion>
</AccordionGroup>

## Performance Considerations

<AccordionGroup>
  <Accordion title="Pre-aggregated data">
    Session data is pre-aggregated, making queries fast even over long time ranges.
  </Accordion>
  
  <Accordion title="Filter by project_id">
    Always include project_id filter for performance and data isolation.
  </Accordion>
  
  <Accordion title="Use appropriate time buckets">
    Query at the granularity you need - hourly for detailed analysis, daily for trends.
  </Accordion>
  
  <Accordion title="Limit release cardinality">
    When querying across many releases, use LIMIT or filter to specific releases.
  </Accordion>
</AccordionGroup>
